# 🧩 Orden final recomendado (roadmap oficial)

### **FASE 1 — núcleo del lenguaje**

1. Parser: 
	Javascript/Typescript
	Python
	Java	
2. Serializador JSON + pretty-printer
3. CLI mínima (en node.js)

### **FASE 2 — utilidad real**

4. Validador de schemas
5. Extensión VSCode
6. Playground

### **FASE 3 — ecosistema**

7. Serializadores a otros formatos
8. Generador de documentación
9. Linter semántico

# ✅ **1. Parser oficial (core STXT-SPEC) — obligatorio**

El parser es la base para absolutamente todo lo demás.

Debe implementar **100% del STXT-SPEC**:

* indentación estricta
* mezcla espacios/tabs prohibida
* nodos `:`
* nodos `>>`
* comentarios fuera de bloques
* normalización (nombres, inline, bloques)
* errores formales

**Lenguajes prioritarios:**

1. **JavaScript / TypeScript**

   * Porque sirve para web, para Node, y para integraciones.
2. **Python**

   * Para automatizaciones, pipelines, IA, APIs.
3. **Java**

   * Importante en enterprise.

Después: C#, Go, Rust.

**En cada versión:**

* parsear → entregar árbol canónico → sin validación semántica.

---

# ✅ **2. Serializador (canonical JSON / stxt pretty-print)** — imprescindible

Un parser sin serializador no permite test unitarios reproducibles.

Necesitas:

### 2.1. JSON canónico

Basado en lo que ya definiste:

```json
{
  "documents": [
      { "name": "Documento", "namespace": "@x", "children": [...] }
  ]
}
```

* siempre orden estable
* sin espacios sobrantes
* sin claves opcionales vacías

Sirve para:

* comparar dos parsers
* test cross-language
* herramientas de CI

### 2.2. Pretty-printer stxt (opcional pero muy útil)

Transforma el árbol parseado en stxt formateado uniformemente.

Sirve para:

* estilizar documentos
* editor / extension VSCode
* auto-format

---

# ✅ **3. Validador de schemas (STXT-SCHEMA-SPEC)** — muy prioritario

Una vez el parser existe, el validador es el segundo gran pilar.

Debe implementar:

* carga de schemas
* asociación por namespace
* tipos (TEXT, NUMBER, BASE64, CODE, etc.)
* cardinalidades
* modo **strict** y modo **non-strict**
* regla 7.1 obligatoria
* errores precisos con línea y tipo

Este componente es el que transforma stxt en un lenguaje *útil para datos reales* (similar a JSON Schema, pero human-first).

---

# ✅ **4. CLI oficial (`stxt`)** — altísimo valor práctico

Una herramienta de línea de comandos que haga:

* `stxt parse file.stxt` → JSON canónico
* `stxt validate file.stxt` → aplica schema
* `stxt format file.stxt` → pretty-print
* `stxt check file.stxt` → lint básico + errores de indentación

Esto también alimenta:

* CI/CD
* Makefiles
* integración con editores
* scripts automáticos

Idealmente escrito en Node.js (cross-platform inmediato) o Rust (portabilidad + velocidad + binarios pequeños).

---

# ✅ **5. Extensión VSCode** — imprescindible a nivel de adopción

Incluye:

### Mínimo viable:

* coloreado (syntax highlighting)
* plegado de bloques (folding de jerarquía)
* reconocimiento básico de `:` y `>>`

### Fase 2:

* autocompletado de nombres según schema
* validación en tiempo real
* tooltip con tipo / cardinalidad
* auto-format

Esta herramienta multiplica la adopción del lenguaje.

---

# ✅ **6. Playground web**

Una web interactiva:

* editor con stxt a la izquierda
* JSON canónico a la derecha
* validación con schema opcional
* indicadores de error en tiempo real

Ideal ejemplo: “JSON Editor Online” pero para stxt.

Esto es fantástico para documentación y demos.

---

# ✅ **7. Libraries para serializar stxt a otros formatos (opcional pero útil)**

* Serializador stxt → **YAML**
* Serializador stxt → **JSON**
* Serializador stxt → **Markdown**
* Serializador stxt → **HTML**

La más esencial es stxt → JSON (ya cubierto), pero un par de extras pueden dar mucho valor en algunos contextos.

---

# ✅ **8. Generador de documentación a partir de schemas**

Similar a:

* JSON Schema → documentación HTML
* OpenAPI → swagger UI

Con un schema stxt, generar:

* tabla de nodos
* cardinalidades
* tipos
* descripciones
* ejemplos de uso

Muy útil para publicar APIs semánticas basadas en stxt.

---

# ✅ **9. Linter semántico (encima del validador)**

Reglas de estilo:

* evitar nombres ambiguos
* evitar nodos vacíos sin necesidad
* recomendación sobre orden de nodos
* guías de estilo corporativo

Esto ya no es esencial para lanzamiento, pero sí útil en equipos grandes.

Template:

Mini lenguaje de templates:

Separador ini: {{ (permitimos cambiar a otra cosa, como  @@ o << ?
Separador end: }} (permitimos cambiar a otra coas como   @@ o >> ? 

Tienen la forma: {{op1:op2:op3:...opn}}

op1: name_op[param opcional]

Las operaciones retornan o bien texto o bien un nodo o un listado de nodos y se van concatenando las operaciones.
Las operaciones DEBEN terminar en un texto.

----------------------------------------
Operaciones básicas que DEBE implementar
----------------------------------------

text: Retorna el texto de un nodo. Aplicable a nodo. Si se aplica a un string retorna el mismo string
children: Retorna los hijos
name: Retorna el nombre (aplicable a un nodo)
name[a name]: Filtra por 'a name' de un listado de nodos
type: Retorna el tipo de un nodo. Aplicable a nodo.
type[a type]: Filtra por 'a type' de un listado de nodos
subtype: Retorna el subtipo de un nodo. Aplicable a nodo
subtype[a subtype]: Filtra por el subtipo de un listado de nodos
render: Renderiza el listado y concatena con ''
render[',']: Renderiza el listado y concatena con
to_upper_case: Aplica a texto
to_lower_case: Aplica a texto
escape_html: Escapa como html
escape_xml: Escapa como xml
escape_js: Escapa como javascript
escape_attrib: Escapa como atrib de xml (quizá otro nombre?)

to_canonic_html???

-------------------------------------------
Operaciones comunes que 'PUEDE' implementar
-------------------------------------------

markdown: Aplica a un texto (NO a un nodo!) y retorna un markdown. Si no está disponible retorna el mismo texto

Ejemplos:

{{text}}	Hola Mundo!
{{type}}    NUMBER, LANG
{{type:subtype}}  markdown
{{children}}  Iteramos childs
{{children:name[apellido]:render}}  Iteramos childs sólo con nombre 'apellido'
{{child:a name}}   Sólo un child (si sólo hay uno, si hay más error)
{{text:markdown}}       Text a markdown
{{text:escape_html}} Text escapado
